Given it's a Node.js + Socket.io chat backend, there are some non-obvious constraints that affect every layer of the setup.

The Critical Socket.io Constraint

Socket.io maintains persistent WebSocket connections. When Auto Scaling adds instances, a client connected to Instance A cannot be routed to Instance B mid-session. This means:

ALB must have sticky sessions (stickiness by cookie) so each client always hits the same instance

Socket.io needs a Redis adapter so events (messages, typing, presence) broadcast across all instances — otherwise users on different instances won't see each other's messages

Without the Redis adapter, auto scaling will silently break real-time features.

Recommended Architecture

Route 53     └── ALB (sticky sessions enabled, SSL termination)          ├── EC2 Instance 1  ──┐          ├── EC2 Instance 2  ──┤── ElastiCache Redis (Socket.io adapter + sessions)          └── EC2 Instance N  ──┘                                 └── MongoDB Atlas (or DocumentDB)  Auto Scaling Group     Min: 2  |  Desired: 2  |  Max: 10

EC2 Configuration

Instance type to start with: t3.medium (2 vCPU, 4GB RAM)

Node.js is single-threaded but you'll run it with PM2 in cluster mode (one process per vCPU), so 2 vCPUs is the minimum meaningful unit

Socket.io keeps connections in memory — 4GB handles ~10k concurrent connections comfortably before you need to scale out

Use t3 not t2 — t3 has unlimited CPU burst by default, better for chat spikes

AMI: Amazon Linux 2023 (smallest footprint, free security patches, AWS-optimized)

Storage: 20GB gp3 (gp3 is cheaper and faster than gp2, no need for more since logs should ship to CloudWatch)

Auto Scaling Group Setup

Launch Template settings:

AMI: Amazon Linux 2023

Instance type: t3.medium (override with t3.large as second priority in mixed instances policy)

User data script: pulls your latest build from S3/ECR and starts PM2

Scaling policies — use Target Tracking, not simple step scaling:

MetricTargetWhyCPU utilization60%Leaves headroom before it actually gets slowALB RequestCountPerTarget~500 req/targetGood for HTTP traffic burstsCustom: Socket.io connection count via CloudWatch~3000/instanceMost relevant for your use case

Scale-out cooldown: 120s — fast enough to handle sudden traffic spikes Scale-in cooldown: 300s — conservative, prevents thrashing during bursty chat activity

Scheduled scaling (if you know your traffic pattern, e.g. 9am–6pm workday):

Scale out to min=4 at 08:45 AM weekdays Scale in  to min=2 at 06:15 PM weekdays

ALB Configuration

Stickiness: Enable, duration = 1 day (matches a typical user session)

Health check path: GET /health — make sure your backend has this endpoint returning 200

Idle timeout: 4000 seconds (ALB default is 60s which will kill WebSocket connections)

Protocol: HTTPS on 443, redirect HTTP → HTTPS

Supporting Services

ElastiCache Redis (cache.t3.micro to start):

Socket.io adapter: @socket.io/redis-adapter

Also use it for rate limiting and session storage

Single-AZ is fine to start, enable Multi-AZ when you go production-critical

What NOT to put on EC2:

MongoDB → use Atlas or DocumentDB (managing Mongo on EC2 with auto scaling is painful)

File uploads → S3 directly from client with presigned URLs

Logs → CloudWatch Logs agent, never write to disk on auto-scaled instances

The One Code Change Needed

Before any of this works, add the Redis adapter to your Socket.io server:

import { createAdapter } from '@socket.io/redis-adapter'; import { createClient } from 'redis';  const pubClient = createClient({ url: process.env.REDIS_URL }); const subClient = pubClient.duplicate(); await Promise.all([pubClient.connect(), subClient.connect()]);  io.adapter(createAdapter(pubClient, subClient));

Without this, two users on different instances will never receive each other's messages regardless of how well the infrastructure is configured.
